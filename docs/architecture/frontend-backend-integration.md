# Frontend–backend integration baseline (DEV-005)

## Scope and evidence

This document records repository evidence as of DEV-005; it does not declare the business UI integrated. The audit covered `frontend/src`, frontend configuration and scripts, `backend/app/main.py`, module registration, authentication dependencies, backend contract tests, and the FastAPI OpenAPI schema generated by the application.

## Audited state

- Authentication in `frontend/src/api/auth.ts` is the only end-to-end frontend workflow whose paths and schemas align with the backend (`POST /api/v1/auth/login`, form data, and authenticated `GET /api/v1/auth/me`). `frontend/src/main.tsx` restores the token, resolves the user, redirects anonymous users, and applies client-side role visibility. Backend authorization remains authoritative.
- `frontend/src/modules/organizations/api.ts` and `frontend/src/modules/registries/api/registries.ts` can switch to in-memory demo arrays through `VITE_DEMO_MODE`; their screens are not unconditionally integrated.
- `RegistriesPage.tsx` is hardcoded display data. Incoming registry uses the conditional API. Several other pages use local state/placeholders.
- `ShipmentsPage.tsx`, `ScannerPage.tsx`, and `SearchPage.tsx` use the Axios instance formerly defined in `frontend/src/shared/api/client.ts`. That boundary used another token key (`access_token`) and assumed the configured URL already included `/api/v1`; these calls therefore did not share the active auth session. Their paths and frontend shapes are not proven against the generated contract.
- Before DEV-005, `frontend/src/api/client.ts`, `frontend/src/api/auth.ts`, and `frontend/src/shared/api/client.ts` duplicated URL/request logic. Defaults disagreed (`localhost` versus `127.0.0.1`, API root versus server root). The ignored local `frontend/.env` is environment-specific state and is not a deployment contract; `.env.example` is the documented template.
- The package had no test or lint script. The production build (`tsc -b && vite build`) was the only configured type check.

## Contract gaps

The backend contract uses the `/api/v1` root, OAuth2 password form login, bearer JWTs, and `{detail: ...}` errors (including FastAPI's 422 issue array). Protected module routes depend on `get_current_user`; some add role dependencies. Pagination and filtering are endpoint-specific rather than one universal envelope. Frontend domain types and demo records frequently use simplified fields and must be mapped per endpoint. Client-side role checks are navigation behavior, not security. Demo credentials and the demo token do not authenticate with FastAPI, and demo mode must never be represented as integration.

`backend/app/schemas/auth.py` and the generated `app__schemas__auth__UserRead` OpenAPI component define `id`, `username`, `full_name`, `role`, nullable `department`, and `is_active`. FastAPI response serialization includes `department` as a field whose value may be `null`; the frontend `User` contract therefore requires `department: string | null` and `is_active: boolean`. The demo user follows the same shape and uses `department: null`.

## Canonical boundaries

`frontend/src/api/client.ts` is the canonical request and configuration boundary. `VITE_API_URL` is required and must be an absolute HTTP(S) URL whose normalized path is exactly `/api/v1` (for example `https://host/api/v1`). An optional supplied trailing slash is removed. Credentials, query strings, fragments, other protocols, and other paths are rejected. Missing or malformed configuration throws `ApiConfigurationError` before any network call; it is not converted to `ApiError(kind: "network")`, and there is no machine-specific fallback.

The client attaches the `pk_skdk_token` bearer token unless a caller explicitly supplies an unauthenticated token provider (login), requests JSON, parses successful JSON, returns `undefined` for 204, and throws `ApiError` for request failures. HTTP errors retain safe backend `detail`, status, and kind. Network errors are normalized without propagating low-level text. Callers own user-facing Ukrainian messages and state handling; they must not log tokens or request authorization headers. Backend 401 clears the session at the authentication orchestration boundary; 403 is an authorization failure, not a logout.

`frontend/src/pages/LoginPage/LoginPage.tsx` is the only caller of the concrete `login()` function. Its caller-level mapping presents only `ApiError(kind: "http", status: 401)` as `Невірний логін або пароль`. Network, configuration, 403, 422, other HTTP, and unknown failures remain distinct from invalid credentials and receive separate Ukrainian messages. The canonical client retains structured errors and does not contain UI-message policy.

Client tests use the Node built-in test runner and Node's TypeScript type-stripping support, avoiding another frontend testing dependency. `frontend/package.json` requires Node `>=22.6.0`, the first Node 22 release with this capability, and CI selects Node 22. The test script retains the explicit `--experimental-strip-types` flag for compatibility across supported Node 22 minors.

The older Axios boundary remains temporarily for three legacy module pages and is deprecated by this decision; it must not be used by new code. Migrating those workflows requires endpoint-specific contract work and is outside DEV-005.

## First vertical slice: authenticated current-user status (DEV-006)

DEV-006 connects and explicitly presents the authenticated-session status in the application shell using `GET /api/v1/auth/me` through the canonical client. In non-demo operation it uses the real login token and maps exactly to `UserRead` (`id`, `username`, `full_name`, `role`, nullable `department`, `is_active`). Successful JSON is validated at runtime before it becomes authenticated UI state. Demo mode remains an intentional mock path and is not evidence of backend integration.

The root authentication orchestrator exposes loading and authenticated states, passes the validated user to the shell, and redirects anonymous or expired sessions to login. Only an HTTP 401 clears the stored token. HTTP 403 preserves the token and renders an authorization-specific status instead of weakening backend-authoritative access or logging the user out. Malformed success responses, network failures, configuration failures, server errors, and unexpected failures also preserve the token and render distinct retry/logout status UI. There is no meaningful empty collection state: a successful response is one user, while a missing or expired session is 401.

The shell presents the backend-provided display name, username, role, and optional department. Session requests use a shared generation guard for both UI updates and token-clearing side effects: only an active `401` may clear the session, and login retries or logout invalidate older requests. Focused tests cover the `UserRead` validator, state/error mapping, rendering text model, generation-safe session clearing, and login/application destinations. A backend contract test fixes both the complete `/auth/me` response shape and its unauthenticated 401 boundary.

## Increment boundaries, risks, and next acceptance criteria

In scope for DEV-005: evidence-based audit, canonical configuration/fetch boundary, auth migration to it, deterministic client tests, and roadmap/architecture documentation. Out of scope: business-module integration, UI redesign, API generation, global state libraries, backend behavior changes, auth bypasses, and normalization of every frontend model.

Operational prerequisites remain a migrated database with an active user, a valid CORS origin for the deployed frontend, secure HTTPS deployment, a supplied `VITE_API_URL`, and demo mode disabled outside intentional demos. Automated coverage does not replace a manual deployed-browser check with a real backend session.
